HTTP Response Splitting


נכנסנו לאתר ‎/doomle.html וגילינו שקלט המשתמש נשלח כערך לפרמטר שנקרא
course_id ל- cgi-bin/course_selector. 
בניתוח תעבורה ב-Wireshark ראינו שהקלט מוזרק ישירות ב- Header שנקרא Demo
בתשובת השרת,
מה שמאפשר לבצע ‎HTTP Response Splitting‎.
אנחנו שולחות שתי בקשות HTTP לשרת. 
הראשונה מכילה את ה-Payload מוזרק בערך הפרמטר,  והשניה היא בקשה רגילה לדף
הקורבן /67607.html.
ה- Payload שלנו הוא בעצם בקשת HTTP שמכילה בערך הפרמטר course_id קוד
שמכיל שלושה חלקים: סיום התשובה הראשונה, התשובה השניה המזויפת, 
ולבסוף סיום הבקשה הראשונה.
כדי לגרום לתשובה הראשונה להסתיים ולאפשר לתשובה המזויפת להופיע כתשובה
נפרדת (ולא כחלק מה-body), השתמשנו ב- Header שנקרא
transfer-encoded: chunked, המאפשר לתוכן ההודעה לקבוע את סיומה.
התשובה המזויפת מוחזרת כתשובת השרת ל-proxy והוא בתורו מכניס אותה לקאש
ומחזיר אותה כתשובה ל- attacker לבקשתו לאתר הקורבן. 
כתוצאה מכך, הדף המורעל מוחזר לכל משתמש המבקר באתר הקורבן.






Blind SQL injection
הקוד מממש מתקפת Blind SQL injection המשתמש באלגוריתם חיפוש בינארי כדי
לחלץ ביעילות נתונים משרת הנתקף, וזאת על בסיס חילוץ ביט מידע מכל תשובה של 
השרת וניתוח תשובות true או false . 
ביצענו בדיקה ידנית מקדימה שבה גלשנו לאתר וזיהינו כי ערך החיפוש של ההזמנה
 משתקף ישירות ב-URL כפרמטר order_id (למשל order_id=2), 
ממצא שאפשר להבין ממנו את מבנה בקשת ה-GET ולבנות את ה- payload הזדוני בהתאם.
ההזרקה עצמה מתבצעת על ידי החדרת שאילתת UNION SELECT זדונית לתוך
 פרמטר ה-order_id, תוך הישענות על העובדה שערך ה-0 עבור order_id בשאילתה
 המקורית אינו מחזיר תוצאות ולכן מאפשר לתנאי המוזרק (הנבדק באמצעות פונקציות
 ASCII ו-SUBSTR) לחלץ את התנהגות השרת המשתנה בהתאם לאמיתות התנאי.
 האלגוריתם מחלץ את סכמת הטבלה שממנה אנו רוצים לגנוב את הסיסמה (שם הטבלה
 והעמודות שלה) ולאחר מכן מתאפשר חילוץ מלא של סיסמת המשתמש הספציפי.
 הסיסמה שחולצה נשמרת ונכתבת אל תוך קובץ.
